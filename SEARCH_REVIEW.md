**审查日期**: 2026-06-24

## 一、功能审查（Functionality）

### ✅ 已修复的问题

#### P0 - 关键BUG
1. **异常堆栈丢失** (`search_engine.py:169`)
   - **问题**: `raise e` 会丢失原始堆栈跟踪，导致调试困难
   - **修复**: 改为 `raise`（保留完整堆栈）
   - **状态**: ✅ 已修复

#### P1 - 重要BUG
2. **`top_k` 参数未验证** (`search_engine.py:304`)
   - **问题**: 如果 `top_k` 为 0 或负数，Qdrant 查询可能失败
   - **修复**: 添加验证，确保 `top_k` 在 1-100 范围内
   - **状态**: ✅ 已修复

### ⚠️ 待优化的问题

#### P1 - 安全风险
3. **HTML 注入风险** (`search_engine.py:849`)
   - **问题**: LLM 返回的 `synthesis` 直接插入 HTML，未转义
   - **风险**: 如果 LLM 被攻破或返回恶意内容，可能执行 XSS
   - **建议**: 
     - 方案A：先 HTML 转义，再还原 LaTeX 公式（复杂）
     - 方案B：使用白名单标签过滤（如 `bleach` 库）
     - 方案C：保持现状（本地应用，风险较低）
   - **状态**: ⏳ 待讨论

#### P2 - 代码质量
4. **死代码/位置不当的代码**
   - **问题**: `_extract_json_block()` 定义在 `search_engine.py`，但实际使用在 `classify_pipeline.py`
   - **建议**: 移至 `utils/` 共享工具模块
   - **状态**: ⏳ 待优化

5. **模块过大** (`search_engine.py` 1219 行)
   - **问题**: 单一文件包含搜索、LLM 调用、HTML 渲染、PDF 生成、分面统计等多个职责
   - **建议**: 拆分为多个模块：
     - `search_core.py` - 搜索核心
     - `llm_client.py` - LLM API 调用
     - `report_renderer.py` - HTML/PDF 渲染
     - `facet_stats.py` - 分面统计
   - **状态**: ⏳ 待重构

6. **缺少类型标注**
   - **问题**: 函数缺少类型标注，降低代码可读性和 IDE 支持
   - **建议**: 添加类型标注（Python 3.10+ 语法）
   - **状态**: ⏳ 待优化

## 三、实现优雅性审查（Implementation Elegance）

### ⚠️ 需要优化的地方

1. **单一职责原则违反**
   - `search_engine.py` 包含多个不相关功能
   - **建议**: 按功能拆分模块（见上文）

2. **硬编码常量**
   - `OUTPUT_DIR`, `LLM_BASE_URL` 等常量定义在模块级别
   - **影响**: 测试困难，不支持运行时配置
   - **建议**: 改为函数参数或配置对象

3. **错误处理不一致**
   - 有些函数返回 `{"ok": False, "error": "..."}` 
   - 有些函数直接抛出异常
   - **建议**: 统一错误处理策略

4. **文档字符串不完整**
   - 部分函数缺少 docstring
   - 部分 docstring 与实际参数不一致
   - **建议**: 补充完整的 docstring（Google 风格）

## 四、测试覆盖审查

### ⚠️ 缺少测试

1. **无单元测试**
   - 搜索阶段核心函数（`search()`, `answer()`, `_query_qdrant_rrf()` 等）无单元测试

   - **建议**: 添加 pytest 单元测试

2. **无集成测试**
   - 未测试搜索 → LLM → HTML 报告的端到端流程
   - **建议**: 添加集成测试（使用 mock LLM API）

## 五、性能审查

### ✅ 性能优化点

1. **分面统计使用滚动查询** (`get_facet_stats()`)
   - 边 scroll 边聚合，不积累全量 points 到内存 ✅

2. **搜索结果去重** (`_dedup_chunks()`)
   - 按 source 分组，保留表格版，丢弃乱码 ✅

### ⚠️ 潜在性能问题

1. **HTML 报告生成无缓存**
   - 每次调用 `answer()` 都重新生成 HTML
   - **建议**: 对相同 query 的结果进行缓存（TTL 1 小时）

2. **图片 base64 编码**
   - 大图片会导致 HTML 文件过大
   - **建议**: 添加图片大小限制或压缩
