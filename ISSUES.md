## v0.4.2（2026-06-17）— PATCH 修复版本

### [#I001] answer() 中 LLM 配置永远为空
- 优先级: 🔴 P0 阻断
- 状态: 🔵 Fixed
- 发现日期: 2026-06-17
- 版本: v0.4.2
- 影响: 勾选"使用 AI 回答"后始终显示"未配置 LLM API"，无法生成回答和 HTML 报告
- 根因: answer() 使用模块级常量 LLM_API_KEY，该常量在 import kb_query 时就已求值（此时 .env 未加载）
- 修复: answer() 和 auto_classify() 改为 os.environ.get("KB_LLM_API_KEY") or LLM_API_KEY

### [#I002] WebSocket 超时导致 connect lost
- 优先级: 🟡 P1 严重
- 状态: 🔵 Fixed
- 发现日期: 2026-06-17
- 版本: v0.4.2
- 影响: 搜索时出现"connect lost"，然后恢复，显示"无回答"
- 根因: NiceGUI 默认 reconnect_timeout=3.0s，但 LLM API 调用需 10-30s
- 修复: ui.run() 设置 reconnect_timeout=120

### [#I003] 系统状态一直显示离线
- 优先级: 🟡 P1 严重
- 状态: 🔵 Fixed
- 发现日期: 2026-06-17
- 版本: v0.4.2
- 影响: 页面顶部始终显示"离线"徽章，用户无法判断系统是否正常
- 根因: (1) QDRANT_URL 使用 "localhost" 解析为 IPv6 ::1，Qdrant 只监听 IPv4 127.0.0.1；(2) 状态徽章 hardcode 为"离线"
- 修复: QDRANT_URL 改为 127.0.0.1；状态徽章动态初始化 + 每 10 秒定时刷新

### [#I004] 报告链接 file:// 浏览器安全阻止
- 优先级: 🟢 P2 一般
- 状态: 🔵 Fixed
- 发现日期: 2026-06-17
- 版本: v0.4.2

- 根因: 按钮用 `file://D:\...` 协议打开本地文件，浏览器安全策略阻止

## v0.5.1（2026-06-20）— 阶段一验证发现（已在 v0.6.0 重构中解决）

> ⚠️ 以下问题在阶段一用户验收过程中发现，属于阶段二（UI 交互重构）范畴。
> **v0.6.0 阶段二重构中通过代码替换全部解决**（非 Bug 修复，而是整段代码重写）。

### [#I016] AI 分析按钮绑定了两个 click handler
- 优先级: 🔴 P0 阻断
- 状态: 🔵 Fixed (v0.6.0)
- 发现日期: 2026-06-20
- 版本: v0.5.1
- 影响: 每次点击 AI 分析触发两次 LLM 调用，浪费资源且容易超时
- 根因: NiceGUI 的 `.on_click()` 是追加 handler 不是替换。行 569 `ai_btn.on_click(on_ai_analyze)` 和行 786 `ai_btn.on_click(do_ai_analyze)` 都被注册了，两个函数都会执行
- 修复: v0.6.0 重构中 `on_ai_analyze` 整个函数删除，`do_ai_analyze` 重写为唯一 handler

### [#I016a] 页面出现两组重复的分面分类下拉菜单 + 两个入库按钮
- 优先级: 🔴 P0 阻断
- 状态: 🔵 Fixed (v0.6.0)
- 发现日期: 2026-06-20
- 版本: v0.5.1
- 影响: 
  - 页面原本只有下半组下拉菜单（主表单：内容类型/主题域/时效属性/认知验证/工作流阶段/关联项目/可信度/关键词 + 1个"摄入到知识库"按钮）
  - 点击 AI 分析后，`on_ai_analyze` 弹出 confirm_card，里面**新建了第二组** 4 个分面下拉菜单 + "✅ 确认并入库"按钮
  - `do_ai_analyze` 试图填充主表单的下拉菜单，但用户被 confirm_card 遮挡/分散，看到的是 AI 结果填进了"新出现的上半组"，而非页面原本的"下半组"
  - 结果：用户面前同时有 2 组分面下拉菜单 + 2 个入库按钮，完全无法判断操作哪个
- 根因: 
  1. I016 双重绑定的直接后果
  2. 设计上 `on_ai_analyze` 用 confirm_card 创建了一套独立的 UI 组件（包含自己的下拉菜单和按钮），而不是复用主表单已有的组件
  3. `do_ai_analyze` 走的是正确路线（填充主表单），但因为两个 handler 都执行，confirm_card 的出现盖过了 do_ai_analyze 的填充效果
- 修复: v0.6.0 重构中 confirm_card 相关代码全部删除，只保留主表单 + 唯一的"摄入到知识库"按钮

### [#I017] do_ai_analyze 不传 metadata，L2 管道（文件元数据→UDC）未生效
- 优先级: 🟡 P1 严重
- 状态: 🔵 Fixed (v0.6.0)
- 发现日期: 2026-06-20
- 版本: v0.5.1
- 影响: 上传文件时提取的元数据（标题、作者等）未传给 auto_classify，L2 推断管道形同虚设
- 根因: `do_ai_analyze` 行 749 调用 `auto_classify(ingest_content)` 未传 metadata 参数；被覆盖的 `on_ai_analyze` 行 484 正确传了 `metadata=meta_for_ai` 但已失效
- 修复: v0.6.0 中 `classify_document()` 签名含 `file_metadata` 参数，`do_ai_analyze` 传入 `STATE.get("auto_metadata", {})`

### [#I018] do_ingest 中 AI 结果合并逻辑是死代码
- 优先级: 🟢 P2 一般
- 状态: 🔵 Fixed (v0.6.0)
- 发现日期: 2026-06-20
- 版本: v0.5.1
- 影响: 代码行 662-666 尝试将 AI 分类结果合并进 metadata，但因 metadata 已包含所有分面字段，条件 `if k not in metadata` 永远为 False，合并永远不执行
- 根因: metadata 在行 647-660 已从下拉菜单取了全部字段，后续合并逻辑条件不可能满足
- 修复: v0.6.0 中 `do_ingest()` 整段重写，删除无效合并逻辑

### [#I019] AI 分析结果不可复现——同一内容每次置信度不同、domain 不稳定
- 优先级: 🔴 P0 阻断
- 状态: 🔵 Fixed (v0.6.0)
- 发现日期: 2026-06-20
- 版本: v0.5.1
- 影响:
  - 同一段文本，每次点击 AI 分析，LLM 返回的 confidence 分数不同
  - domain 字段有时有值、有时为空、有时结果不一致
  - 置信度路由（≥0.8 直接入库 / 0.5-0.8 needs_review / <0.5 死信队列）形同虚设——同一内容可能这次直入库、下次进死信队列
- 根因（三重）:
  1. **LLM temperature=0.3**（`_call_llm_api` 行 1919）— 非确定性采样，相同输入产生不同输出
  2. **置信度由 LLM 生成而非程序计算** — v0.4.5 设计决策明确写了"启发式 (JSON完整性 0.25 + 字段合法性 0.35 + 信息丰度 0.25 + 一致性 0.15)"，但代码中从未实现，直接用 LLM 自报的 confidence 值
  3. **L2/L3 关键词映射表仅 6 条**（行 2033-2040）— 绝大多数文本无法被规则命中，全部落入 L4 LLM 路径
- 修复: v0.6.0 三层管道全部解决：
  - temperature → 0（确定性输出）
  - `calculate_confidence()` 程序计算置信度（替代 LLM 自报）
  - `CLASSIFY_RULES` 规则引擎扩充至 40+ 关键词 + 正则模式
