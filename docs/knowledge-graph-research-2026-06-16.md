# 知识图谱深度研究报告

> 研究日期：2026-06-16
> 研究目的：为 Athanor 故事一（知识关系网）设计自研方案，不照搬竞品代码
> 研究对象：微软 GraphRAG / LightRAG（港大） / nano-graphrag / RAGFlow 知识图谱

---

## 一、背景与动机

### 1.1 为什么需要知识图谱

传统向量检索的局限：
- **语义相似 ≠ 逻辑相关**：「齿轮强度」和「材料力学」语义距离远，但逻辑上强相关
- **关系缺失**：向量只捕获内容相似度，无法表达「A 反驳 B」「A 派生自 B」
- **检索结果孤立**：无法呈现知识条目的关联网络

### 1.2 Athanor 已有的基础（独特优势）

其他项目做知识图谱是**从零开荒**（文档 → LLM 提取实体 → 建图），Athanor 已有：

| 基础 | 说明 | 图谱价值 |
|------|------|----------|
| 8 种关系类型已定义 | `similar/references/contradicts/derived_from/supports/questions/answers/related` | 可直接作为边类型建图 |
| 分面分类 4 维度已标注 | `content_type/domain/temporal_nature/epistemic_status` | 可作为节点属性 + 可视化着色 |
| 知识条目已结构化 | 36 字段，每条有 `relations` 字段 | 零 LLM 成本建图 |
| Qdrant 向量库已在用 | 向量相似检索已成熟 | 图谱 + 向量混合检索 |

→ **Athanor 的知识图谱不是从零开荒，而是从已有关系字段自动建图 + 按需 LLM 补全**

## 二、三大流派对比

### 2.1 微软 GraphRAG

**来源**：微软研究院，2024 年发布

**核心流程**：
```
全部文档
  → LLM 实体+关系提取（数百次调用）
  → Leiden 社区发现
  → LLM 社区摘要（每个社区一段自然语言摘要）
  → 层次化社区（高层 = 宏观摘要，底层 = 微观实体）
```

**检索方式**：
- 用户输入查询 → 匹配到相关社区 → 返回社区摘要（而非原始分块）
- 适合**全局问答**（「这个数据集的整体主题是什么？」）

**成本**：
- 建图：数百次 LLM 调用（对 100 万字文档，约 $15-30 美元）
- 更新：全量重建（不支持增量）

**评价**：
- ✅ 检索质量高（尤其全局问答）
- ❌ 成本极高
- ❌ 不适合个人知识库（规模小，不需要社区摘要）

---

### 2.2 LightRAG（香港大学）

**来源**：港大研究团队，2024 年 arxiv 论文

**核心改进**（vs GraphRAG）：
- **增量更新**：新文档到来时，只提取新增实体和关系，不重建全图
- **双层检索**：Local（实体-specific）+ Global（跨社区）
- **更低成本**：建图只需几次 LLM 调用（vs GraphRAG 的数百次）

**流程**：
```
文档分块
  → LLM 提取实体+关系（单次调用，输出 JSON）
  → 去重合并（规则，无 LLM 成本）
  → 向量库存储实体描述和关系
  → 检索时：实体链接 → 图遍历 → 向量重排序
```

**代码规模**：约 5000 行 Python

**评价**：
- ✅ 增量更新（适合持续摄入的个人 KB）
- ✅ 成本可控
- ❌ 仍需 LLM 提取（个人场景可按需触发，不必全量）

---

### 2.3 nano-graphrag

**来源**：个人开源项目（GitHub gusye1234/nano-graphrag）

**定位**：GraphRAG 的**精简学习版**（1100 行 Python）

**核心设计**：
- 保留 GraphRAG 的核心思路（实体提取 → 社区发现 → 社区摘要）
- 移除企业级复杂性（多租户、权限、分布式）
- 适合**学习实验**（理解 GraphRAG 原理）

**评价**：
- ✅ 代码简洁，易理解
- ❌ 仍是全量建图（无增量）
- ❌ 社区活跃度一般

---

### 2.4 综合对比表

| 流派 | 复杂度 | LLM 成本 | 增量更新 | 适合 |
|------|:--:|:--:|:--:|------|
| 微软 GraphRAG | 高（数万行） | 极高（数百次调用） | ❌ | 企业全局问答 |
| LightRAG（港大） | 中（5000 行） | 低（几次调用） | ✅ | 持续更新的知识库 |
| nano-graphrag | 低（1100 行） | 中 | ❌ | 学习实验 |
| **RAGFlow** | **中** | **低（Light 模式）** | **✅** | **个人 + 企业** |
| **Athanor（规划）** | **低** | **极低（按需，非全量）** | **✅** | **个人知识引擎** |

## 三、RAGFlow 知识图谱详细分析

### 3.1 架构（无 Neo4j）

```
文档上传
  → 分块（按解析器类型）
  → LLM 实体+关系提取（单次调用）
  → LLM 实体消解（二次调用）
  → NetworkX 建图（内存）
  → Leiden 社区发现
  → LLM 社区摘要（三次调用）
  → 序列化为 GEXF 持久化
```

**关键发现**：RAGFlow **没有用 Neo4j**，而是用 NetworkX（Python 图论库）+ GEXF 序列化。这验证了 Athanor 选 NetworkX 的方向正确。

### 3.2 双模式

| 模式 | LLM 调用次数 | 速度 | 质量 |
|------|-------------|------|------|
| Light | 1 次（仅实体提取） | 快 | 基础图谱 |
| General | 3 次（提取+消解+摘要） | 慢 | 完整图谱 |

**Athanor 应优先实现 Light 模式**（零/低 LLM 成本），General 模式留远期。

### 3.3 关系标签机制

RAGFlow **不预设固定关系类型**，由 LLM 自由发现：
- LLM 输出三元组 `(entity, relation, entity)`，relation 是自然语言标签
- 可视化时，边按 relation 类型着色/区分
- **检索价值**：图遍历时沿特定关系类型扩展（如「只找反驳关系」）

**Athanor 现状**：8 种预设关系类型（手动 + LLM 推断），无自由发现。

## 四、Athanor 自研方案

### 4.1 设计原则

1. **非必要不用大模型**（架构原则第 1 条）— 能用固定程序完成的，不调 LLM
2. **从已有数据建图** — `relations` 字段已有关系，直接建图，零 LLM 成本
3. **按需 LLM 补全** — 不批量全量提取，用户触发时才调 LLM
4. **分面分类天然着色** — Athanor 独有优势，竞品无此能力

### 4.2 三步实施方案

#### 第一步：零 LLM 建图（v0.5.0，故事一）

```
Qdrant 全量读取
  → 每条知识条目 = 图节点
  → node_id = doc_id
  → node 属性 = content_type / domain / epistemic_status / temporal_nature / title
  → relations 字段 = 图边
  → edge 属性 = relation_type（8 种预设）
  → NetworkX 建图
  → 序列化为 GEXF（或 JSON）持久化
```

**成本**：零 LLM 调用。

**难点**：当前 `relations` 字段的数据覆盖率（有多少条目真的填了 relations？）。如覆盖率低，第一步图谱会很稀疏。

**缓解方案**：
- 第一步先做可视化（即使稀疏也有价值）
- 第二步按需 LLM 补全（见下）

---

#### 第二步：按需 LLM 关系发现（v0.5.x）

**触发时机**（非全量，按需）：
1. 用户查看某条目详情 → 自动发现与该条目相关的其他条目关系
2. 用户主动点击「发现关系」按钮
3. 摄入新文档后，自动发现与已有知识的关系

**LLM 调用设计**（参考 RAGFlow Light 模式）：
```
输入：条目 A 的文本 + Top-K 相似条目（向量检索）的文本
输出：JSON 数组，每条 `[entry_B_doc_id, relation_type, confidence]`
```

**relation_type 处理**：
- 优先匹配预设 8 种（精确匹配）
- 无法匹配 → 自由标签（存为 `free_relation_tag` 字段）
- 高频自由标签 → 升级为预设关系类型（人工确认）

**置信度路由**（复用 v0.4.5 机制）：
- ≥ 0.8 → 自动写入 `relations` 字段
- 0.5–0.8 → 入库但标记 `needs_review`
- < 0.5 → 进审核队列

---

#### 第三步：分面联动检索（v0.6.0+）

**当前检索**：向量相似 + 分面过滤（两个独立维度）

**增强后检索**：向量相似 + 分面过滤 + **关系图谱遍历**（三个维度）

**场景示例**：

**场景 1**：「找反驳我的设计决策的已验证知识」
```
epistemic_status=["corroborated"]
  → 过滤出已验证知识
  → 在图谱中找与用户设计文档有 `contradicts` 关系的条目
  → 返回结果
```

**场景 2**：「找与这篇论文相关的所有知识，按领域分面」
```
论文条目 → 图谱遍历（1-2 跳）
  → 找出所有相关条目
  → 按 domain 分面聚合
  → 返回分面统计 + 条目列表
```

**场景 3**：「图谱引导探索」
```
图谱社区检测（Leiden）
  → 发现用户在「机械设计」领域有 15 个 `contradicts` 关系
  → 推荐用户探索该领域的矛盾知识
```

---

### 4.3 可视化方案

#### 后端：NetworkX

- 零依赖（已内置在 Python 标准数据科学生态）
- 图算法丰富（最短路径、社区发现、中心性）
- 可序列化为 GEXF / GraphML / JSON

#### 前端：PyVis 或 D3.js

**方案 A — PyVis**（快速实现）：
- Python 库，生成交互式 HTML（`<div id="graph">`）
- 内置力导向布局（ForceAtlas2）
- 可自定义节点颜色/大小/标签
- **NiceGUI 集成**：`ui.html(pyvis.get_html())` 嵌入

**方案 B — D3.js**（定制化强）：
- JavaScript 库，需自己写前端代码
- 可视化效果更精美
- 交互更灵活（缩放/平移/筛选）
- **NiceGUI 集成**：`ui.add_body_html()` 注入 D3.js 脚本

**推荐路径**：先 PyVis（快速验证），后 D3.js（定制化）

#### 分面着色方案（Athanor 独特优势）

| 视觉通道 | 映射字段 | 说明 |
|---------|---------|------|
| 节点颜色 | `domain`（UDC 9 主类） | 一眼看出跨领域关联 |
| 节点边框 | `epistemic_status`（L0/L1/L2） | L0 虚线 / L1 单线 / L2 双线 |
| 节点大小 | `trust_score` | 可信度高的条目节点更大 |
| 边颜色 | `relation_type` | `contradicts` 红 / `supports` 绿 / `references` 灰 |
| 边线型 | `relation_type` | `contradicts` 虚线 / `supports` 实线 |

## 五、与竞品的边界（不做什么）

| 不做的事 | 理由 |
|---------|------|
| ❌ 社区发现 + 摘要（GraphRAG 风格） | 个人 KB 规模不需要，LLM 成本高 |
| ❌ 引入 Neo4j / 外部图数据库 | NetworkX 零依赖足够，个人场景无需图查询语言 |
| ❌ 从零开荒式实体提取 | Athanor 分面分类 + 已有 relations 字段已定义知识维度 |
| ❌ 批量全量 LLM 关系提取 | 成本高，个人场景应按需触发 |
| ❌ 完整的图谱编辑器 | 过度设计，查看 + 高亮足矣 |
| ❌ 多跳推理（逻辑推理链） | 属于 Crucible（炼真）职责范围，不放在 Athanor |

## 六、实施路径（详细版本）

```
v0.5.0  故事一：知识关系网（基础图谱）
  ├── 从 relations 字段自动构建 NetworkX 图谱（零 LLM）
  ├── 图谱可视化（PyVis，NiceGUI ui.html 嵌入）
  ├── 分面着色（domain 颜色 + epistemic_status 边框）
  ├── 基础交互（点击节点查看详情弹窗）
  └── GEXF 序列化持久化

v0.5.x  故事一增强：LLM 关系发现（按需）
  ├── 按需 LLM 关系提取（非全量，Top-K 相似条目）
  ├── 关系标签归一化（自由标签 → 预设 8 种映射表）
  ├── 置信度路由集成（复用 v0.4.5 机制）
  └── 审核队列支持（低置信度关系进审核队列）

v0.6.0  故事二：QA 生成摄入（并行开发）
  ├── 文档 → LLM 拆分为问答对
  ├── QA 对转标准 chunk 格式入库
  └── 摄入管线「启用 QA 拆分」可选开关

v0.6.x  关系标签联动检索
  ├── 分面 + 图谱联合检索（三维检索）
  ├── 关系类型作为第三检索维度
  ├── 图谱引导分面探索（社区检测 → 推荐过滤条件）
  └── D3.js 可视化升级（替代 PyVis）

v0.7.0  故事三：管线配置化
  ├── 摄入步骤 YAML 声明
  ├── 步骤可配置/可跳过/可调参
  └── DAG 并行摄入（多文件）

v0.8.0  异步摄入 + 守望文件夹
  ├── 双队列架构（处理队列 + 死信队列）
  ├── 守望文件夹自动摄入
  └── 实时进度推送（WebSocket）

v1.0.0  生产就绪
  ├── 插件协议（MCP / OpenAPI）
  ├── 工作流可视化编排（可选）
  ├── 桌面端一键打包
  └── 移动端适配
```
