## 一、RAGFlow — 知识图谱与精细化解析

### 1.1 知识图谱（核心差异化功能）

**架构**：NetworkX 内存图 + 向量DB双存储（**未用 Neo4j**），图谱序列化为 GEXF 格式持久化。

**管线**（LLM 调用链）：
```
文档分块
  → LLM 实体+关系提取（单次调用，输出三元组）
  → LLM 实体消解（二次调用，"AI"="人工智能"合并）
  → Leiden 社区发现（图算法，无LLM成本）
  → LLM 社区摘要（三次调用，为每个社区生成自然语言摘要）
```

**双模式**：
| 模式 | LLM 调用 | 适合 |
|------|---------|------|
| Light | 1次（仅实体提取） | 个人快速预览 |
| General | 3次（提取+消解+摘要） | 高质量图谱 |

**关系标签机制**：
- 不预设固定关系类型，由 LLM **自由发现**并打标签（"用于""导致""反驳"等自然语言标签）
- 图谱可视化中，边按关系类型着色/区分
- **检索价值**：图遍历检索（从查询实体出发沿边扩展 Top-K）+ 社区摘要检索

**Athanor 可借鉴**：
- ✅ Light 模式思路（零/低 LLM 建图）— 可先做零 LLM 版（从已有 relations 字段建图）
- ✅ 图谱可视化嵌入 Web UI
- ❌ 不照搬 General 模式（社区摘要 LLM 成本过高，个人 KB 不需要）

---

### 1.2 文档解析（7种解析器矩阵）

RAGFlow 的核心优势之一是**精细化文档解析**，支持：

| 解析器 | 适合 | 特点 |
|--------|------|------|
| Naive | 纯文本 | 最快，无结构 |
| General | 通用文档 | 自动分块 |
| QA | 问答对 | **LLM 拆文档为问答对**（FastGPT 也有类似功能） |
| Resume | 简历 | 结构化提取 |
| Paper | 论文 | 公式/图表/引用保留 |
| Book | 书籍 | 章节结构保留 |
| Tag | 标签文档 | 元数据丰富 |

**与 Athanor 的关系**：
- `content_type` 字段已有 15 种类型，解析器选型可参考此矩阵
- QA 解析器 = FastGPT QA 生成模式的 RAGFlow 实现（见第三节）

---

### 1.3 其他值得注意的功能

- **Agent 融合**：RAGFlow 支持 Agent 模式（自动选择工具/检索策略）
- **企业级成熟度**：多用户、权限管理、API 完善
- **7种解析器**：覆盖场景广，但部分与 Athanor 的 `content_type` 分面有重叠

## 二、Dify — 工作流引擎与管线配置化

### 2.1 四层解耦架构

```
编排层：YAML DSL 定义管线步骤
执行层：Python 沙箱执行（安全隔离）
数据流：DAG 有向无环图（并行调度）
会话层：三级状态存储（Redis / PostgreSQL / OSS）
```

**DAG 并行调度**：3.2 倍提速（vs 串行执行）

**Python 沙箱安全限制**：
- CPU 10s 超时
- 内存 256MB 上限
- 网络白名单
- 只读文件系统

### 2.2 可视化工作流编辑器

- 节点拖拽式编排（类似 n8n / Node-RED）
- 每个节点可配置模型、提示词、参数
- 支持条件分支、循环、并行

**Athanor 可借鉴**：
- ✅ **摄入管线 YAML 配置化**（学 Dify 编排层，但不需要完整可视化编辑器）
- ✅ **DAG 并行摄入**（多文件并行处理）
- ❌ 不照搬可视化编辑器（过度设计，YAML 配置足够）

### 2.3 插件市场与生态

- 60k+ ⭐ 社区
- 插件市场（工具/模型/数据源）
- 监控与日志

## 三、FastGPT — QA生成与异步队列

### 3.1 QA 自动生成（⭐ 最直接可学）

**核心思路**：
```
文档 → LLM 拆分为问答对 → 问答对向量化入库 → 检索时直接匹配问题
```
**效果**：检索质量优于直接分块（问答对更贴合用户搜索意图）

**实现细节**（基于 FastGPT 源码分析）：
- LLM 调用参数：`temperature=0.3`，`stream=false`，`timeout=10min`
- 输出格式统一：QA 对转成标准 chunk 格式入库（与直接分块格式一致）
- **模式统一**是关键设计：检索时不区分「原文分块」和「QA分块」

**Athanor 可借鉴**：
- ✅ 作为摄入管线的**可选开关**（「启用 QA 拆分」勾选框）
- ✅ 嵌入现有两阶段摄入管线（阶段二 AI 分析时可选触发）
- 预估工作量：~200 行代码

---

### 3.2 双队列异步架构

```
文件上传 → 入队（MongoDB）
          ↓
      Worker 1（OCR）     Worker 2（嵌入）     Worker 3（入库）
          ↓                   ↓                    ↓
      完成回调 → 状态更新 → 前端实时进度
```

**MongoDB 分布式锁**：多 Worker 互不干扰

**Athanor 可借鉴**：
- ✅ 双队列（处理队列 + 死信队列）— v0.4.5 已规划死信队列
- ✅ Worker 异步摄入 — v0.8.0 规划

---

### 3.3 API 熔断机制

**问题**：LLM API 调用中途余额不足 → 部分数据已处理、部分未完成 → 状态不一致

**FastGPT 方案**：
```
任务开始 → 检查余额（API 余额查询）
         ↓ 余额不足
         → 锁定任务（标记 status=blocked）
         → 通知用户充值
         ↓ 余额充足
         → 正常执行
```

**Athanor 可借鉴**：
- ✅ 在「引擎配置」页面增加余额检查（DeepSeek API 支持余额查询）
- ✅ 任务锁定 + 恢复机制

### 5.2 三个故事的优先级排序

基于「低投入高回报」原则：

| 优先级 | 故事 | 学自 | 难度 | 预计版本 |
|--------|------|------|------|---------|
| 1 | 知识关系网（Light图谱） | RAGFlow | 🟡 中 | v0.5.0 |
| 2 | QA 自动生成摄入 | FastGPT | 🟢 低 | v0.6.0 |
| 3 | API 熔断机制 | FastGPT | 🟢 低 | v0.4.5 穿插 |
| 4 | 摄入管线 YAML 配置化 | Dify | 🟢 低 | v0.7.0 |
| 5 | LLM 关系发现（按需） | RAGFlow | 🟡 中 | v0.5.x |
| 6 | 双队列异步摄入 | FastGPT | 🟡 中 | v0.8.0 |

---

### 5.3 不做什么（与竞品边界）

- ❌ **社区发现+摘要**（GraphRAG 风格）— 个人 KB 规模不需要，LLM 成本高
- ❌ **Neo4j / 外部图数据库** — NetworkX 零依赖足够
- ❌ **完整可视化工作流编辑器** — 过度设计，YAML 配置足够
- ❌ **多用户/权限管理** — 一人公司场景
- ❌ **插件市场** — v1.0.0 之后考虑
